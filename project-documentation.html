Core Entities & Relationships

Users (1) ←→ (Many) Documents Users (1) ←→ (Many) WorkFlows (as reviewers) Documents (1) ←→ (Many) WorkFlows Documents (1) ←→ (Many) AuditLogs Users (1) ←→ (Many) AuditLogs

Database Tables Structure

Table Name Purpose Key Fields Relationships
users Store user information and roles id, name, email, password, role One-to-Many with Documents, WorkFlows, AuditLogs
documents Store document metadata and file paths id, title, upload_path, uploaded_by, status, created_at Many-to-One with Users, One-to-Many with WorkFlows
workflows Track approval workflow steps id, document_id, reviewer_id, status, comment, reviewed_at Many-to-One with Documents and Users
audit_logs Track all system activities id, action, entity_type, entity_id, user_email, timestamp References Users and Documents

Entity Class Details

User Entity

@Entity @Table(name = "users") public class User { @Id @GeneratedValue private int id; private String name, email, password; @Enumerated(EnumType.STRING) private Role role; @OneToMany(mappedBy = "uploadedBy") private List<Document> documents; @OneToMany(mappedBy = "reviewer") private List<WorkFlow> reviews; }

Document Entity

@Entity @Table(name = "documents") public class Document { @Id @GeneratedValue private Long id; private String title, uploadPath; @ManyToOne @JoinColumn(name = "uploaded_by") private User uploadedBy; @OneToMany(mappedBy = "document") private List<WorkFlow> workFlow; @Enumerated(EnumType.STRING) private Status status; }

🔄 Document Approval Workflow

Multi-Level Approval Process

1
Document Upload
User uploads document → System creates workflow with 3 approval levels
↓
2
Level 1: Officer Review
Officer receives notification → Reviews document → Approves/Rejects
↓
3
Level 2: Manager Review
If approved → Manager receives notification → Reviews → Approves/Rejects
↓
4
Level 3: Admin Review
If approved → Admin receives notification → Final review → Approves/Rejects
↓
5
Completion
Document status updated → Notifications sent → Audit trail logged

Workflow Creation Logic

private void createDefaultWorkflow(Document document) { // Find users by role User officer = userRepository.findFirstByRole(Role.OFFICER); User manager = userRepository.findFirstByRole(Role.MANAGER); User admin = userRepository.findFirstByRole(Role.ADMIN); // Create workflow steps WorkFlow step1 = new WorkFlow(0, document, officer, Status.PENDING, null, null); WorkFlow step2 = new WorkFlow(0, document, manager, Status.PENDING, null, null); WorkFlow step3 = new WorkFlow(0, document, admin, Status.PENDING, null, null); // Save all workflow steps workFlowRepository.saveAll(List.of(step1, step2, step3)); }

Status Flow

PENDING → UNDER_REVIEW → APPROVED / REJECTED

🔐 Security Implementation

Authentication Flow

  1. User submits credentials (email/password)
  2. System validates against database
  3. JWT token generated with user role
  4. Token returned to client
  5. Client includes token in subsequent requests

Security Features

  • JWT Tokens: Stateless authentication
  • BCrypt: Password encryption
  • Role-based Access: Method-level security
  • CORS: Cross-origin resource sharing
  • Input Validation: Request sanitization

📧 Notification System

Email Notifications

  • Document Upload: Notify first reviewer (Officer)
  • Approval: Notify next reviewer in chain
  • Rejection: Notify document uploader
  • Completion: Final status notification

Audit Trail

Every action is logged with:

  • Action performed
  • User who performed it
  • Timestamp
  • Entity affected
  • IP address

📊 Database Schema & Relationships

🔗 Key Relationships Explained

One-to-Many: A user can upload multiple documents, but each document has only one uploader.

Many-to-Many: Documents go through multiple workflow steps with different reviewers.

Audit Trail: Every action creates an audit log entry linking user, entity, and action.

Complete Database Schema

-- Users table CREATE TABLE users ( id SERIAL PRIMARY KEY, name VARCHAR(255) NOT NULL, email VARCHAR(255) UNIQUE NOT NULL, password VARCHAR(255) NOT NULL, role VARCHAR(20) NOT NULL ); -- Documents table CREATE TABLE documents ( id BIGSERIAL PRIMARY KEY, title VARCHAR(255) NOT NULL, upload_path VARCHAR(500) NOT NULL, uploaded_by INTEGER REFERENCES users(id), created_at TIMESTAMP NOT NULL, status VARCHAR(20) NOT NULL, comment TEXT, reviewed_at TIMESTAMP ); -- Workflows table CREATE TABLE workflows ( id SERIAL PRIMARY KEY, document_id BIGINT REFERENCES documents(id), reviewer_id INTEGER REFERENCES users(id), status VARCHAR(20), comment TEXT, reviewed_at TIMESTAMP ); -- Audit logs table CREATE TABLE audit_logs ( id BIGSERIAL PRIMARY KEY, action VARCHAR(100) NOT NULL, entity_type VARCHAR(50) NOT NULL, entity_id BIGINT NOT NULL, user_email VARCHAR(255) NOT NULL, user_name VARCHAR(255) NOT NULL, details TEXT, timestamp TIMESTAMP NOT NULL, ip_address VARCHAR(45) );

Sample Data Flow

Example: Document Upload Process

  1. User uploads document: Creates record in documents table
  2. Workflow creation: Creates 3 records in workflows table (Officer, Manager, Admin)
  3. Audit logging: Creates record in audit_logs table
  4. Email notification: Sends email to first reviewer (Officer)
  5. Review process: Updates workflow status and creates audit logs
